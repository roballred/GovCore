# Reference Source: TOGAF Standard

> This is reference material, not a GovEA capability definition. GovEA capability definitions in this folder govern product direction when there is a conflict.

## Source

- **Name:** TOGAF Standard
- **Maintainer:** The Open Group Architecture Forum
- **Current baseline for GovEA planning:** TOGAF Standard, 10th Edition, including Technical Corrigendum 1
- **Primary source:** <https://publications.opengroup.org/standards/togaf>
- **Overview:** <https://www.opengroup.org/togaf>

## Why It Matters To GovEA

Some government Enterprise Architects and central IT organizations use TOGAF language to describe architecture method, governance, architecture content, and roadmap planning. GovEA should be able to align to that language when it helps with credibility, procurement, audit, or enterprise coordination.

GovEA should not require TOGAF knowledge for ordinary users. Department Directors, contributors, and small agency teams should continue to see plain-language GovEA concepts first.

## GovEA Interpretation

| TOGAF Area | GovEA Alignment |
|---|---|
| Architecture Development Method | Optional ADM phase alignment over existing GovEA content |
| Business Architecture | Capabilities, personas, value streams, services, strategic objectives |
| Application Architecture | Application portfolio and application-to-capability traceability |
| Technology Architecture | Future technology portfolio and standards/principles support |
| Architecture Content | GovEA records, relationships, ADRs, principles, glossary, and reports |
| Enterprise Architecture Capability and Governance | IAM, audit trail, review workflow, ADRs, principles, and future governance capabilities |
| Business Capabilities and Value Streams guidance | Existing capability map and value stream capabilities |
| Migration Planning | Initiatives, roadmap views, and future migration/reporting outputs |

## Planning Constraints

- TOGAF is an optional overlay, not a replacement for EasyEA.
- GovEA should map to TOGAF concepts rather than reproduce TOGAF as a full workflow engine.
- Framework language should be hidden from non-architect views unless explicitly enabled.
- Any use of TOGAF text must respect The Open Group's licensing and citation requirements.
