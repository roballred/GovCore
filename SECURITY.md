# Security Policy

GovEA is an open-source enterprise architecture tool used by state and local government practitioners. The data it stores — application portfolios, capability maps, architecture decisions — is operationally sensitive even when not classified. We take security reports seriously and appreciate the work of researchers who report issues responsibly.

## Reporting a Vulnerability

**Please do not open a public GitHub issue for security vulnerabilities.**

Use one of these private channels instead:

1. **GitHub Private Vulnerability Reporting** (preferred) — open the [Security tab](https://github.com/roballred/GovEA/security/advisories/new) on this repository and click *Report a vulnerability*. This creates a private advisory only the maintainers can see.

2. **Email** — if GitHub Private Vulnerability Reporting is unavailable to you, contact the maintainer through the email listed on their GitHub profile.

## What to Include

A useful report contains, at minimum:

- A clear description of the vulnerability and the impact
- Step-by-step instructions to reproduce
- The affected version, branch, or commit hash
- Any proof-of-concept code, requests, or screenshots
- Your suggested severity (we'll calibrate, but your view is helpful)
- Whether you would like credit and how you would like to be named

If you can include a suggested fix, that's welcome but not required.

## Response Expectations

GovEA is an open-source project without a paid security team. We aim for the following best-effort response times:

| Action | Target |
|---|---|
| Initial acknowledgment | Within 3 business days |
| Triage decision (in scope / out of scope / severity) | Within 7 business days |
| Fix or mitigation for confirmed Critical/High issues | Within 30 days |
| Coordinated public disclosure | Within 90 days of report (sooner with reporter agreement) |

If you do not receive an acknowledgment within 3 business days, please follow up — the message may have been missed.

## Disclosure Policy

We follow a **90-day coordinated disclosure** model:

- Once a fix is available, we will publish a GitHub Security Advisory and credit the reporter (unless they prefer anonymity)
- We may request a longer embargo for complex issues; we will not extend embargoes unilaterally
- If we cannot reach agreement on disclosure timing, the reporter retains the right to disclose 90 days after the initial report

If you are evaluating GovEA for use in a government environment and need to perform a security assessment, that is permitted on **your own deployment only** — please do not test against community deployments or any infrastructure you do not own.

## In Scope

The following are in scope for security reports:

- The GovEA application code in this repository (`apps/govea/`, `packages/`)
- The default deployment configurations (`docker/`, `Containerfile`)
- Documented installation and operational guidance that, if followed, leads to an insecure deployment
- Build artifacts published from this repository

We are particularly interested in:

- Authentication and authorization bypasses
- Cross-tenant data exposure (multi-tenant isolation failures)
- SQL injection, command injection, server-side request forgery
- Stored XSS or HTML injection that survives the markdown / URL allowlists
- Privilege escalation between roles (Viewer → Contributor → Admin → Instance Admin)
- Audit-log integrity issues
- Secrets leakage (logs, error pages, repository content)
- Vulnerabilities in our dependency tree that materially affect GovEA users

## Out of Scope

The following are **not** in scope and we will not act on reports limited to them:

- Issues in third-party services that GovEA does not control (your IdP, your hosting provider, etc.)
- Social engineering of contributors or users
- Denial-of-service attacks that depend on volume rather than a code defect
- Self-XSS (where the user is required to paste an attacker-supplied payload into their own browser)
- Missing security headers without a demonstrated impact
- Reports generated by automated scanners without a working proof of concept
- Theoretical vulnerabilities that require physical access, a compromised endpoint, or a malicious package already running on the server
- Issues in code paths that are clearly marked as `dev-only` or `tests/`

If you are unsure whether something is in scope, send the report — we will make the call.

## Safe Harbor

We will not pursue legal action against researchers who:

- Report vulnerabilities through the private channels above
- Make a good-faith effort to avoid privacy violations, data destruction, and service disruption
- Do not exploit a vulnerability beyond what is necessary to demonstrate the issue
- Give us reasonable time to fix and disclose before going public

This is a good-faith commitment, not a legal contract. If you have concerns about how a specific test plan would be received, ask first.

## Acknowledgments

We will acknowledge reporters in the relevant Security Advisory and, with permission, in the project's public credits. Anonymous reports are honored — say so in your initial message.

## Security Hardening Status

Active security work is tracked under the [`security` label](https://github.com/roballred/GovEA/issues?q=is%3Aopen+label%3Asecurity) on this repository. The [v0.9 — Foundation Cleanup milestone](https://github.com/roballred/GovEA/milestone/1) currently includes the highest-severity findings from a recent internal scan. Until those land, operators should treat any deployment as a development build.
